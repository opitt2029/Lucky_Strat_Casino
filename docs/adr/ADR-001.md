# ADR-001 — 資料庫分配：PostgreSQL（寫入主庫）+ MySQL（查詢讀庫）

| 項目 | 內容 |
|------|------|
| **狀態** | ✅ 已接受（Accepted） |
| **日期** | 2026-05-26 |
| **決策者** | 全員（組長 A 主導） |
| **影響範圍** | Wallet Service、Member Service、Game Service、Rank Service |

---

## 背景

幸運星幣城平台需要同時處理兩種性質截然不同的資料操作：

1. **帳務核心操作**（高一致性需求）：星幣扣款、派彩入帳、餘額異動。這些操作若出錯（例如重複扣款或超扣），將直接影響使用者體驗，且難以修復。
2. **高頻查詢操作**（高讀取效能需求）：帳務明細分頁、玩家列表查詢、排行榜歷史查詢。這些操作讀多寫少，對一致性要求相對寬鬆，但對回應速度要求高。

若只使用單一資料庫，兩種需求會互相競爭資源，且難以針對各自特性做最佳化。

---

## 決策選項

### 選項 A — 單一 PostgreSQL

```
所有讀寫 ──► PostgreSQL
```

- 優點：架構最簡單，不需要資料同步
- 缺點：讀取效能受寫入鎖競爭影響；無法針對查詢場景獨立調優；單點風險

### 選項 B — 單一 MySQL

```
所有讀寫 ──► MySQL
```

- 優點：架構簡單，組員最熟悉
- 缺點：MySQL 的樂觀鎖支援（FOR UPDATE SKIP LOCKED）相較 PostgreSQL 弱；ACID 保證在高併發情況下較不穩定

### 選項 C — PostgreSQL（寫）+ MySQL（讀）CQRS 分離 ✅ 已選擇

```
寫入操作 ──► PostgreSQL（帳務主庫）
查詢操作 ──► MySQL（讀庫，由 Kafka 事件同步）
```

- 優點：
  - 寫入端（PostgreSQL）享有強 ACID、精準 Row-Level Lock、可靠樂觀鎖
  - 讀取端（MySQL）針對查詢優化，讀寫不互相干擾
  - 清楚的 CQRS（命令查詢職責分離）邊界，未來可獨立擴展
- 缺點：
  - 架構複雜度提升，需維護兩套資料庫連線
  - 讀庫與寫庫之間存在**最終一致性**延遲（毫秒至秒級）
  - 學習成本較高，需要組員理解 CQRS 概念

---

## 決策

**採用選項 C**：PostgreSQL 作為帳務核心寫入主庫，MySQL 作為查詢讀庫。

### 決策理由

1. **樂觀鎖的可靠性是關鍵**  
   Wallet Service 的 `wallets.version` 樂觀鎖是防止高併發超扣的核心機制（T-022）。PostgreSQL 對 `UPDATE ... WHERE version = ?` 的 Row-Level Lock 實作比 MySQL 更精準可靠，在 1,000 玩家同時下注的壓測情境（T-090）下更重要。

2. **帳務資料不能有讀寫鎖競爭**  
   帳務查詢（T-025）是高頻操作，若和扣款/入帳的寫入鎖競爭同一張 Table，會拖慢回應速度。分離到 MySQL 讀庫後，查詢效能獨立，不受寫入鎖影響。

3. **符合微服務資料所有權原則**  
   每個 Service 對自己的資料庫負責，讀寫分離進一步強化這個邊界。

## 資料庫分配明細

### PostgreSQL（寫入主庫）

| Table | 所屬 Service | 說明 |
|-------|-------------|------|
| `wallets` | Wallet | 餘額、凍結金額、樂觀鎖 version |
| `wallet_transactions` | Wallet | 帳務流水（write 端） |
| `game_rounds` | Game | 遊戲對局完整紀錄 |
| `rank_history` | Rank | 週排行榜歷史快照 |
| `rank_daily_snapshots` | Rank | 每日持幣量快照 |
| `game_rtp_stats` | Game | RTP 統計（每小時排程寫入） |
| `admin_alerts` | Admin | 異常告警紀錄 |

### MySQL（查詢讀庫）

| Table | 所屬 Service | 說明 |
|-------|-------------|------|
| `members` | Member | 玩家帳號（唯一寫入端，不是讀庫同步） |
| `friendships` | Member | 好友關係 |
| `daily_checkins` | Member | 簽到紀錄 |
| `task_definitions` | Member | 任務定義 |
| `player_tasks` | Member | 玩家任務進度 |
| `gift_logs` | Wallet | 好友贈幣紀錄 |
| `wallet_transactions` | Wallet | 帳務流水（read 端，供 T-025 查詢分頁） |

> **注意**：`wallet_transactions` 在兩個資料庫各有一份。PostgreSQL 是寫入來源；MySQL 的讀庫版本由 Wallet Service 在寫入 PostgreSQL 後同步寫入（雙寫策略），或透過 Kafka 事件驅動同步。

## 後果與注意事項

### 開發者需要了解的事

1. **查詢 API 走 MySQL，寫入 API 走 PostgreSQL**  
   Wallet Service 的 `GET /api/v1/wallet/transactions`（T-025）使用 MySQL DataSource；  
   `POST /internal/wallet/debit`（T-022）使用 PostgreSQL DataSource。

2. **最終一致性延遲**  
   讀庫的資料比寫庫晚幾毫秒到幾秒。`GET /api/v1/wallet/balance` 為確保最新餘額，**不走讀庫快取，直接查 PostgreSQL**（參見 T-021 工作說明）。

3. **Spring Boot 多資料源設定**  
   各 Service 需要設定兩個 DataSource Bean，並用 `@Primary` 和 `@Qualifier` 區分：
   ```java
   // PostgreSQL — 帳務主庫（@Primary）
   @Bean @Primary
   public DataSource postgresDataSource() { ... }

   // MySQL — 查詢讀庫
   @Bean @Qualifier("mysqlDataSource")
   public DataSource mysqlDataSource() { ... }
   ```

4. **Port 使用非預設值**  
   - PostgreSQL：本機 Port `5433`（避免與本機已安裝的 5432 衝突）
   - MySQL：本機 Port `3307`（避免與本機已安裝的 3306 衝突）
