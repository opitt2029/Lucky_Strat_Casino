# ADR-002 — wallet.credit 事件契約：分離「入帳指令」與「入帳事件」

| 項目 | 內容 |
|------|------|
| **狀態** | ✅ 已接受（Accepted） |
| **日期** | 2026-05-29 |
| **決策者** | 組長 A |
| **影響範圍** | Wallet Service、Member Service、Rank Service、Notification Service |

---

## 背景

`wallet.credit` 這個 Kafka topic 在實作過程中被兩種**相反語意**同時使用，造成衝突：

| 來源 | 對 `wallet.credit` 的用法 | 隱含語意 |
|------|--------------------------|----------|
| member-service 簽到 / 新手禮（已上線） | 發出以「觸發」wallet 入帳 | **指令**（請入帳） |
| kafka-init 註解 + 與 `wallet.debit` 對稱 | 入帳後才發 | **事件**（已入帳） |
| rank-service（T-040，未實作） | 消費以更新排行榜 | 預期是**事件** |

衍生兩個問題：

1. **T-017 簽到入帳鏈路斷裂**：member 把 `wallet.credit` 當指令發出，但 wallet-service 沒有任何 consumer 去消費它、真正加餘額 → 玩家簽到後餘額不會增加。
2. **若直接補 consumer 會無限迴圈**：T-023 的 `credit()` 入帳後會發 `wallet.credit`；若 wallet 又消費 `wallet.credit` 入帳，就會「自己發、自己收、再入帳」無限循環、重複加錢。

根因：**指令（command，請做某事）與事件（event，某事已發生）混用同一個 topic**，違反事件驅動設計的基本分離原則。

---

## 決策選項

### 選項 1 — `wallet.credit` 當「指令」
wallet 直接消費 member 發的 `wallet.credit` 入帳。

- 優點：改動最少（只加 consumer）。
- 缺點：T-023 的 HTTP 端點必須改成「不發 wallet.credit」（否則迴圈）；rank 消費的是「指令」而非確認結果，語意模糊；與 `wallet.debit`（事件）不對稱。

### 選項 2 — 入帳一律走 HTTP
member 簽到 / 新手禮改成同步呼叫 `POST /internal/wallet/credit`。

- 優點：只有一條入帳路徑。
- 缺點：**破壞 member 現有的 Outbox 原子性**（簽到記錄與入帳不再同交易）；在 DB 交易中做跨服務同步呼叫是 anti-pattern。

### 選項 3 — 分離指令與事件 topic ✅ 已選擇

```
member 簽到/新手禮 ──發──> wallet.credit.request (指令)
                                  │  wallet 消費 → 呼叫 credit() 真正入帳
                                  ▼
                          ──發──> wallet.credit (事件) ──> rank / notification 消費

game 派彩 ──HTTP──> POST /internal/wallet/credit → credit() → 同樣發 wallet.credit (事件)
```

- 優點：
  - 指令 / 事件清楚分離，`wallet.credit` 與 `wallet.debit` 語意對稱（皆為「已發生」事件）。
  - 保留 member 的 **Outbox 原子性**（非同步、可靠投遞）。
  - **沒有迴圈**：消費的是 `wallet.credit.request`、發出的是 `wallet.credit`，兩個不同 topic。
  - T-023 的 `credit()` **完全重用、不需改動**（HTTP 與 Kafka 兩條入帳路徑共用同一個服務方法）。
- 缺點：新增一個 topic；member 的 outbox topic 名需改；wallet 需加一個 consumer。

---

## 決策

**採用選項 3**：

- 新增指令 topic `wallet.credit.request`（+ DLT `wallet.credit.request.DLT`）。
- `wallet.credit` 自此**僅作為「已入帳事件」**，只由 wallet-service 在入帳後發布。
- **wallet-service 永不消費 `wallet.credit`**（避免迴圈）。

### 事件契約

**topic `wallet.credit.request`（指令）**
- 發布者：member-service（簽到 T-017、新手禮 T-018），未來其他需觸發入帳者亦同。
- 消費者：wallet-service（`WalletCreditRequestListener` → `WalletService.credit()`）。
- Payload（JSON，消費端以 `@JsonIgnoreProperties(ignoreUnknown=true)` 容忍額外欄位）：
  ```json
  { "playerId": 42, "amount": 50, "subType": "CHECKIN",
    "idempotencyKey": "checkin-42-2026-05-29", "referenceId": null }
  ```
  - `subType` 須為 DB 允許的 CREDIT 子型：`WIN/CHECKIN/TASK/GIFT/GM_REWARD/BANKRUPTCY_AID`。
  - `idempotencyKey` 保證 Kafka 重送不重複入帳（DB UNIQUE）。

**topic `wallet.credit`（事件）**
- 發布者：wallet-service（`credit()` 入帳成功後）。
- 消費者：rank-service（T-040 更新排行榜）、notification-service（T-072 等）。

---

## 後果與注意事項

1. **冪等性是安全網**：`wallet.credit.request` 為 at-least-once 投遞，`credit()` 以 `idempotencyKey` 去重，重送指令不會重複加錢。
2. **失敗處理**：消費失敗（格式錯誤直送、暫時性失敗重試耗盡）會進 `wallet.credit.request.DLT`，可供 Admin 檢視重送（呼應 T-028）。
3. **rank-service 實作須知**：T-040 應消費 `wallet.credit` / `wallet.debit`（事件），不要消費 `wallet.credit.request`（指令）。
4. **不要在 wallet-service 消費 `wallet.credit`**：否則與 `credit()` 發出的事件形成迴圈。
